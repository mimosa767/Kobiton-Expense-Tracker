There are three bugs in withKobitonIosEmbedFrameworks in withKobitonSDK.js that are causing the framework embed phase to silently fail. All three are regex anchors that don't match Expo 54 / React Native 0.81 generated pbxproj format.
Bug 1 — PBXFrameworksBuildPhase regex assumes an empty files list:
The regex /isa = PBXFrameworksBuildPhase;[\s\S]*?files = \(\s*\);/ only matches if the files list is completely empty. CocoaPods populates it with its own frameworks, so it's never empty and this step silently skips.
Bug 2 — Hardcoded UUID 13B07F8C1A680F5B00A75B9A doesn't exist:
This is from an old React Native template. Expo 54 / RN 0.81 generates different UUIDs. Step 5 (adding embed phase to PBXNativeTarget buildPhases) silently fails because the anchor string is never found.
Bug 3 — AppDelegate.swift PBXGroup anchor may not match:
Expo 54 uses ExpoAppDelegate — AppDelegate.swift may not appear in the group by that exact name.
The fix: Replace withKobitonIosEmbedFrameworks with the withXcodeProject API from @expo/config-plugins instead of raw pbxproj string manipulation. It provides reliable methods like xcodeProject.addFramework() that work regardless of UUID or file list contents. Here's the replacement:
jsfunction withKobitonIosEmbedFrameworks(config, options) {
  return withXcodeProject(config, (mod) => {
    const xcodeProject = mod.modResults;
    const frameworks = [];
    if (options.biometricSupport) {
      frameworks.push({ name: 'KobitonLAContext.framework', path: 'KobitonFrameworks/KobitonLAContext.framework' });
    }
    if (options.imageInjectionSupport) {
      frameworks.push({ name: 'KobitonSdk.framework', path: 'KobitonFrameworks/KobitonSdk.framework' });
    }
    for (const fw of frameworks) {
      xcodeProject.addFramework(fw.path, {
        customFramework: true,
        embed: true,
        sign: true,
        link: true,
      });
      console.log(`[KobitonSDK] ✓ Embedded ${fw.name} via xcodeProject.addFramework()`);
    }
    return mod;
  });
}
