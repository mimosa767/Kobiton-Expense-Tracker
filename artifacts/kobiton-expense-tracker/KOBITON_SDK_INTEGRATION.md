## Architecture overview

| Concern | iOS | Android |
|---|---|---|
| Biometric interception | `KobitonLAContext.framework` — drop-in `LAContext` replacement | `KobitonBiometric.aar` — drop-in `BiometricPrompt` replacement |
| Image injection | `KobitonSdk.framework` — swizzles `AVCaptureVideoDataOutput` | `camera2.aar` — replaces `android.hardware.camera2.*` |
| Frame capture (QR) | `KobitonCaptureModule.m` — own `AVCaptureVideoDataOutput` session | `KobitonCameraActivity.kt` — own `kobiton.hardware.camera2` session |
| RN native module (biometric) | `KobitonBiometricModule.m` (pure ObjC) | `KobitonBiometricModule.kt` + `KobitonPackage.kt` |
| RN native module (camera) | `KobitonCaptureModule.m` (pure ObjC) | `KobitonCameraModule.kt` + `KobitonPackage.kt` |

All native module code is auto-generated by the Expo config plugin (`plugins/withKobitonSDK.js`) during `expo prebuild`. You never edit these files directly — edit the plugin.

## SDK files to stage

Place the binary SDKs in `sdk-files/` before running `expo prebuild`. The plugin auto-copies them into the native project.

```
sdk-files/
  ios/
    KobitonSdk.framework/         ← image injection
    KobitonLAContext.framework/   ← biometric
  android/
    camera2.aar                   ← image injection
    KobitonBiometric.aar          ← biometric
```

The plugin copies these to:
- `ios/KobitonFrameworks/KobitonSdk.framework`
- `ios/KobitonFrameworks/KobitonLAContext.framework`
- `android/app/libs/camera2.aar`
- `android/app/libs/KobitonBiometric.aar`

Download links (ask your Kobiton account team for access):
- iOS image injection: `https://kobiton.s3.amazonaws.com/downloads/KobitonSDK-ios.zip`
- Android image injection: `https://kobiton.s3.amazonaws.com/downloads/camera2.aar`
- Biometric SDKs: downloaded from the Kobiton portal

## iOS — Biometric SDK

### What it does
`KobitonLAContext.framework` is a **drop-in subclass of Apple's `LAContext`**. When loaded, it intercepts all `evaluatePolicy:localizedReason:reply:` calls so the Kobiton platform can inject pass/fail results remotely during test sessions. No app logic changes are needed beyond swapping the class.

### Plugin automation (happens automatically during `expo prebuild`)
- Copies `KobitonLAContext.framework` from `sdk-files/ios/` → `ios/KobitonFrameworks/`
- Writes `ios/KobitonFrameworks/KobitonLAContext.podspec` (`vendored_frameworks`)
- Injects `pod 'KobitonLAContext', :path => 'KobitonFrameworks'` into the Podfile
- Adds `FRAMEWORK_SEARCH_PATHS` pointing to `$(PROJECT_DIR)/KobitonFrameworks`
- Adds `-ObjC` to `OTHER_LDFLAGS`
- Adds `NSFaceIDUsageDescription` and `NSAppTransportSecurity → NSAllowsArbitraryLoads` to Info.plist
- Writes `KobitonBiometricModule.m` and registers it in the Xcode Sources build phase

### Why pure ObjC for the native module (not Swift)
- Swift's `RCT_EXTERN_MODULE` relies on the ObjC runtime finding a Swift-generated class at bridge startup — fragile, can be dead-stripped.
- Pure ObjC with `+moduleName` + `+load { RCTRegisterModule(self); }` is guaranteed to appear in `NativeModules`.

### Why `performSelector:` for `configure()`
`KobitonLAContext.h` does **not** declare `configure()` in its public header. Swift cannot call undeclared methods statically — the AppDelegate patch fails to compile. ObjC `performSelector:` sends the message at runtime with a `respondsToSelector:` guard. `+load` fires before `AppDelegate`, so the embedded `GCDWebServer` starts before any React Native bridge activity.

### KobitonBiometricModule.m — key design decisions
```objc
// DIRECT import — not NSClassFromString fallback.
// NSClassFromString(@"KobitonLAContext") ?: [LAContext class] is WRONG:
// if the framework is absent it silently falls back to stock LAContext,
// which Kobiton cannot intercept.
#import <KobitonLAContext/KobitonLAContext.h>

// extern void needed for Xcode 26+ stricter clang modules
extern void RCTRegisterModule(Class);

+ (void)load {
    RCTRegisterModule(self);
    SEL configureSel = NSSelectorFromString(@"configure");
    if ([KobitonLAContext respondsToSelector:configureSel]) {
        [KobitonLAContext performSelector:configureSel];
    }
}

RCT_EXPORT_METHOD(authenticate:(NSString *)reason
                  resolve:(RCTPromiseResolveBlock)resolve
                  reject:(RCTPromiseRejectBlock)reject)
{
    KobitonLAContext *ctx = [[KobitonLAContext alloc] init];
    [ctx evaluatePolicy:LAPolicyDeviceOwnerAuthenticationWithBiometrics
        localizedReason:reason
                  reply:^(BOOL success, NSError *error) {
        dispatch_async(dispatch_get_main_queue(), ^{
            resolve(@{@"success": @(success)});
        });
    }];
}
```

### Framework embedding — why CocoaPods podspec (not `addFramework()`)
`xcodeProject.addFramework()` with `embed:true` silently fails on EAS-generated Expo projects: the framework is linked at build time but never copied into the `.ipa`'s `Frameworks/` directory → `dyld` crashes at launch with `Library not loaded: @rpath/KobitonLAContext.framework/KobitonLAContext`.

**The fix**: register each framework as a CocoaPods pod with a minimal podspec (`vendored_frameworks`). CocoaPods then handles embed + sign automatically via the `[CP] Embed Pods Frameworks` build phase already in every Expo project.

## iOS — Image Injection SDK

### What it does
`KobitonSdk.framework` swizzles `AVCaptureVideoDataOutput`'s sample-buffer delegate method. When active in a Kobiton test session, every frame that arrives via `AVCaptureVideoDataOutput` is replaced with the injected image — including the CameraView live preview.

### Plugin automation (happens automatically during `expo prebuild`)
- Copies `KobitonSdk.framework` from `sdk-files/ios/` → `ios/KobitonFrameworks/`
- Writes `ios/KobitonFrameworks/KobitonSdk.podspec`
- Injects `pod 'KobitonSdk', :path => 'KobitonFrameworks'` into the Podfile
- Adds `FRAMEWORK_SEARCH_PATHS` and `-ObjC` linker flag
- Adds `NSCameraUsageDescription` and `KobitonImageInjectionEnabled = YES` to Info.plist

### What you do NOT need to do
- You do NOT need to call any SDK initialization method. `KobitonSdk.framework` self-initialises via `+load` / `+initialize` at dylib load time.
- You do NOT need to change any camera code in JS. The swizzle is transparent to `expo-camera`.

## iOS — KobitonCaptureModule

### Why this exists (the dead ends we hit first)

| Approach | Why it fails |
|---|---|
| `CameraView.takePictureAsync()` | Uses `AVCapturePhotoOutput` — Kobiton does **not** swizzle this output → captures real camera frame, not injected image |
| `captureRef` / `react-native-view-shot` | Uses `UIGraphicsBeginImageContextWithOptions` (software rendering) — cannot capture Metal/GPU-composited layers (`AVSampleBufferDisplayLayer`) → always returns a **black frame** |

### Solution
Mirror the Android approach exactly: open a **separate** `AVCaptureSession` + `AVCaptureVideoDataOutput`. Kobiton swizzles **every** `AVCaptureVideoDataOutput` delegate, so the injected frames arrive in this new output too.

### Flow
1. JS unmounts `CameraView` (releases the `AVCaptureSession` hardware lock)
2. JS calls `NativeModules.KobitonCaptureModule.captureFrame(1500)`
3. Native module creates its own `AVCaptureSession` + `AVCaptureVideoDataOutput`
4. After 1500 ms delay, the module sets `_armed = YES`
5. The next arriving sample buffer (now Kobiton-injected) is captured
6. `CMSampleBuffer` → `UIImage` (rotated to portrait via `UIImageOrientationRight`) → JPEG → base64
7. Promise resolves with the base64 string
8. JS passes base64 to `jpeg-js` + `jsQR` for decoding
9. JS re-mounts `CameraView` after 350 ms

### Critical implementation details
```objc
// CORRECT constant — kCVPixelBufferLockFlags_ReadOnly does NOT exist
CVPixelBufferLockBaseAddress(imageBuffer, kCVPixelBufferLock_ReadOnly);

// Camera frames arrive landscape — rotate to portrait for jsQR
UIImage *raw = [UIImage imageWithCGImage:cgImage scale:1.0
                              orientation:UIImageOrientationRight];

// Weak pointer dereference — capture strong local first
__weak KobitonCaptureModule *weakSelf = self;
dispatch_after(..., ^{
    KobitonCaptureModule *s = weakSelf;   // ← strong local
    if (!s || s->_settled) return;        // ← access via strong local
    s->_armed = YES;
});
```

### Disable `onBarcodeScanned` on iOS
```jsx
<CameraView
  onBarcodeScanned={Platform.OS === 'android' ? handleBarcodeScanned : undefined}
  barcodeScannerSettings={Platform.OS === 'android' ? { barcodeTypes: ['qr'] } : undefined}
/>
```
`onBarcodeScanned` activates `AVCaptureMetadataOutput`. This conflicts with Kobiton's `AVCaptureVideoDataOutput` swizzle → native crash on Kobiton sessions.

## Android — Biometric SDK

### What it does
`KobitonBiometric.aar` is a drop-in replacement for `androidx.biometric`. Import `com.kobiton.biometric.BiometricManager` and `com.kobiton.biometric.BiometricPrompt` instead of the AndroidX versions.

### Prerequisites — app must meet ALL of these
- Targets Android 9 (API 28) or later
- Uses `BiometricPrompt.AuthenticationCallback` or `BiometricPrompt.PromptInfo`
- Does **NOT** use `BiometricPrompt.CryptoObject` (higher-security API — unsupported)
- Does **NOT** use the deprecated `FingerprintManager` API

### Plugin automation
- Copies `KobitonBiometric.aar` → `android/app/libs/`
- Adds `implementation fileTree(dir: 'libs', include: ['*.aar'])` to `build.gradle`
- Adds to `AndroidManifest.xml`:
  - `<uses-permission android:name="android.permission.USE_BIOMETRIC" android:requiredFeature="false"/>`
  - `<uses-permission android:name="android.permission.INTERNET"/>`
  - `android:usesCleartextTraffic="true"` on `<application>`
- Writes `KobitonBiometricModule.kt`, `KobitonCameraModule.kt`, `KobitonCameraActivity.kt`, `KobitonPackage.kt`
- Patches `MainApplication.kt` to register `KobitonPackage()`

### Import replacements
```kotlin
// Before
import androidx.biometric.BiometricManager
import androidx.biometric.BiometricPrompt

// After
import com.kobiton.biometric.BiometricManager
import com.kobiton.biometric.BiometricPrompt
```

### Critical: No Toast in AuthenticationCallback
```kotlin
// WRONG — causes NullPointerException on Kobiton sessions
override fun onAuthenticationSucceeded(result: BiometricPrompt.AuthenticationResult) {
    Toast.makeText(context, "Success", Toast.LENGTH_SHORT).show()  // ← crash
}

// CORRECT — use Log or update UI state from the main thread
override fun onAuthenticationSucceeded(result: BiometricPrompt.AuthenticationResult) {
    Log.d(TAG, "authenticate: succeeded")
    promise.resolve(mapOf("success" to true))
}
```
`Toast` on the callback thread (which may not be the main Looper thread) causes `java.lang.NullPointerException: Can't toast on a thread that has not called Looper.prepare()` during Kobiton sessions.

## Android — Image Injection SDK

### What it does
`camera2.aar` replaces `android.hardware.camera2.*` so the Kobiton platform can intercept and inject synthetic camera frames.

### Plugin automation
- Copies `camera2.aar` → `android/app/libs/`
- Adds `implementation fileTree(dir: 'libs', include: ['*.aar'])` to `build.gradle`
- Adds to `AndroidManifest.xml`:
  - `<service android:name="kobiton.hardware.camera2.ImageInjectionClient"/>`
  - `INTERNET` and `ACCESS_NETWORK_STATE` permissions

### Import replacements (for custom native modules)
```kotlin
// Before
import android.hardware.camera2.CameraManager
import android.hardware.camera2.CameraDevice
import android.hardware.camera2.CameraCaptureSession
import android.hardware.camera2.CaptureRequest
import android.media.ImageReader

// After
import kobiton.hardware.camera2.CameraManager
import kobiton.hardware.camera2.CameraDevice
import kobiton.hardware.camera2.CameraCaptureSession
import kobiton.hardware.camera2.CaptureRequest
import kobiton.media.ImageReader
```

### CameraManager initialization change
```kotlin
// Before
val manager = context.getSystemService(Context.CAMERA_SERVICE) as CameraManager

// After — required for Kobiton injection
val manager: CameraManager = CameraManager.getInstance(context)
```

### Note for expo-camera users
`expo-camera` and `expo-image-picker` use `camera2` internally. The `camera2.aar` intercepts at the OS level, so **JS-layer camera APIs work without import changes** as long as the `.aar` is loaded and `ImageInjectionClient` is registered in the manifest.

## Android — KobitonCameraModule

### Why this exists
`expo-camera`'s `CameraX` uses Android's `camera2` API but `CameraX` does not expose Kobiton-injected frames through its `takePicture()` API reliably. Calling `KobitonCameraActivity` directly (which uses `kobiton.hardware.camera2.CameraManager.getInstance()`) is the reliable path.

### Flow
1. JS unmounts `CameraView` (releases the camera2 hardware resource)
2. JS calls `NativeModules.KobitonCameraModule.openCamera()` → Promise
3. `KobitonCameraModule.kt` launches `KobitonCameraActivity` via `startActivityForResult`
4. `KobitonCameraActivity` opens camera using `CameraManager.getInstance(context)` (Kobiton-intercepted)
5. User taps capture button → JPEG written to `cacheDir` → `file://` URI returned
6. JS receives the URI, reads it as base64 with `FileReader`, decodes with `jpeg-js` + `jsQR`
7. `CameraView` re-mounts after 350 ms

### KobitonCameraActivity registration in AndroidManifest.xml (auto by plugin)
```xml
<activity
  android:name="com.kobiton.expensetracker.KobitonCameraActivity"
  android:screenOrientation="portrait"
  android:theme="@style/Theme.AppCompat.NoActionBar"
  android:exported="false" />
```

## JS layer — calling the native modules

### Biometric authentication
```typescript
import { NativeModules, Platform } from 'react-native';
import * as LocalAuthentication from 'expo-local-authentication';

async function authenticate(reason: string): Promise<boolean> {
  const mod = NativeModules.KobitonBiometricModule;
  if (mod) {
    // Use Kobiton-interceptable native module
    const result = await mod.authenticate(reason);
    return result.success === true;
  }
  // Fallback for dev builds without the native module
  const result = await LocalAuthentication.authenticateAsync({ promptMessage: reason });
  return result.success;
}
```

### QR / image capture
```typescript
import { NativeModules, Platform } from 'react-native';
import jsQR from 'jsqr';
import * as jpeg from 'jpeg-js';

async function uriToBase64(uri: string): Promise<string> {
  const response = await fetch(uri);
  const blob = await response.blob();
  return new Promise((resolve, reject) => {
    const reader = new FileReader();
    reader.onload = () => {
      const result = reader.result as string;
      resolve(result.slice(result.indexOf(',') + 1));
    };
    reader.onerror = reject;
    reader.readAsDataURL(blob);
  });
}

// atob() — works in both Hermes (Android) and JSC (iOS). Buffer is NOT available in Hermes.
async function decodeQRFromBase64(b64: string): Promise<string | null> {
  const binaryStr = atob(b64);
  const bytes = new Uint8Array(binaryStr.length);
  for (let i = 0; i < binaryStr.length; i++) bytes[i] = binaryStr.charCodeAt(i);
  const { data, width, height } = jpeg.decode(bytes, { useTArray: true });
  const code = jsQR(new Uint8ClampedArray(data), width, height);
  return code ? code.data : null;
}

// iOS
async function captureIOS(): Promise<string | null> {
  setCameraVisible(false);  // unmount CameraView first
  const b64 = await NativeModules.KobitonCaptureModule.captureFrame(1500);
  return decodeQRFromBase64(b64);
}

// Android
async function captureAndroid(): Promise<string | null> {
  setCameraVisible(false);
  const uri = await NativeModules.KobitonCameraModule.openCamera();
  const b64 = await uriToBase64(uri);
  return decodeQRFromBase64(b64);
}
```

## Camera timing fix (critical)

### The bug
On Kobiton, camera permission is pre-granted at session launch. If you write:
```typescript
useEffect(() => {
  if (permission?.granted && tab === 'qr') {
    setTimeout(() => setCameraVisible(true), 1500);
  }
}, [permission?.granted]);  // ← BUG: only fires once
```
The 1500 ms timer fires once at app launch. `cameraVisible` becomes `true`. 30+ seconds later the user taps the QR Scanner tab — `CameraView` mounts **instantly** with no warm-up → races with Kobiton's swizzle hooks → crash on first click every time.

### The fix
```typescript
useEffect(() => {
  setCameraVisible(false);  // ← ALWAYS reset on every QR tab visit
  if (permission?.granted && tab === 'qr') {
    const delay = Platform.OS === 'ios' ? 1500 : 350;
    const t = setTimeout(() => setCameraVisible(true), delay);
    return () => clearTimeout(t);
  }
}, [permission?.granted, tab]);  // ← tab in deps ensures re-run on every switch
```
This guarantees a fresh warm-up window every time the user navigates to the QR tab, regardless of when permission was granted.

## Known pitfalls and dead ends

### iOS

| Pitfall | Explanation |
|---|---|
| `takePictureAsync()` returns real camera frame | Uses `AVCapturePhotoOutput` — Kobiton swizzles `AVCaptureVideoDataOutput` only |
| `react-native-view-shot` returns black frame | Software rendering (`UIGraphicsBeginImageContextWithOptions`) cannot capture Metal-composited `AVSampleBufferDisplayLayer` |
| `NSClassFromString(@"KobitonLAContext") ?: [LAContext class]` | Falls back silently to stock `LAContext` if framework is absent — Kobiton cannot intercept stock calls |
| `KobitonLAContext.configure()` in Swift AppDelegate | `configure()` is not in the public header — Swift static call fails to compile |
| `xcodeProject.addFramework(embed: true)` | Framework is linked but not copied into `.ipa` → `Library not loaded` crash at launch |
| `RCT_EXTERN_MODULE` Swift approach | Fragile — class can be dead-stripped by linker; fails with old arch |
| `onBarcodeScanned` enabled on iOS | `AVCaptureMetadataOutput` conflicts with Kobiton's `AVCaptureVideoDataOutput` swizzle → crash |
| `kCVPixelBufferLockFlags_ReadOnly` | Does not exist — correct constant is `kCVPixelBufferLock_ReadOnly` |
| Dereferencing `__weak` ivar pointer directly | Compile error in clang — capture into a strong local first |

### Android

| Pitfall | Explanation |
|---|---|
| `Toast` inside `AuthenticationCallback` | `NullPointerException` — callback thread has no Looper |
| `context.getSystemService(CAMERA_SERVICE)` | Returns stock `CameraManager` — Kobiton cannot intercept |
| `BiometricPrompt.CryptoObject` | Not supported by `KobitonBiometric.aar` |
| `FingerprintManager` | Deprecated, not intercepted |
| `Buffer` API in Hermes | Node.js `Buffer` does not exist in Hermes (Android JS engine) — use `atob()` instead |

### General

| Pitfall | Explanation |
|---|---|
| Concurrent EAS builds from same directory | Run iOS build, wait for it to upload, then run Android |
| Ad Hoc `.ipa` on unregistered device | Device UDID must be in provisioning profile at build time — requires new build per new device |
| pnpm patch for slider not applied on EAS | EAS may reinstall from cache; use a Podfile `post_install` hook to stub `RNCSliderComponentView.mm` instead |
